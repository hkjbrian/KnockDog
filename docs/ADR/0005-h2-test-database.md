# ADR-0005: 테스트 DB로 H2 인메모리 채택

- **상태**: Accepted
- **결정일**: 2026-07-16

## 맥락 (Context)
앱 런타임은 Docker MySQL을 사용한다(→ [ADR-0006](0006-docker-mysql.md)). 하지만 **테스트는 실행 환경에 의존하지 않고 독립적으로** 돌아야 한다 — 테스트 환경과 실행 환경의 분리가 필요하다.

테스트에서도 MySQL을 쓰면, 격리를 지키기 위해 **실행용 MySQL과 별개로 테스트용 DB를 추가로 띄워야** 한다(추가 컨테이너/인스턴스 + 관리).

## 검토한 후보 (Candidates)
- **후보 A — H2 인메모리 (테스트 전용 프로파일)**: 컨테이너 0개, 프로세스 내에서 완결 → 테스트/실행 환경이 완전히 분리되고 가장 가볍다. DB 설치·기동 불필요. MySQL과 방언 차이는 있으나 `MODE=MySQL`로 완화, 스키마도 단순.
- **후보 B — 테스트용 MySQL 별도 기동 (Testcontainers 등)**: 운영과 동일해 정확하지만, 테스트마다 실행용과 별개의 MySQL 컨테이너를 띄워야 해 무겁고 Docker에 의존.
- **후보 C — 실행용 MySQL을 테스트에서 재사용**: 컨테이너는 안 늘지만 테스트가 실행 환경에 결합 → 격리가 깨지고(데이터 오염) DB가 없으면 테스트가 실패.

## 결정 (Decision)
**후보 A(H2 인메모리)를 채택.** 테스트 환경을 실행 환경에서 분리하고, 테스트는 **어떤 컨테이너도 없이** 프로세스 내에서 돌게 한다.

- 앱 런타임(main)은 그대로 MySQL 사용.

## 결과 (Consequences)
- `./gradlew build`/`test`가 **DB·Docker 없이 통과**한다. (`clean build` 그린 확인 완료)
- H2와 MySQL의 방언·타입 차이가 있으므로, MySQL 특화 기능을 쓰게 되면 테스트 신뢰도에 주의. 현재 스키마(users·refresh_tokens)는 단순해 문제없음.
- 운영 DB와 동일한 환경에서의 통합 검증이 필요해지면 **Testcontainers(후보 B)** 도입을 별도 ADR로 재검토한다.
